## Introduction

EventHub — NED is a full-stack event and competition management system built on ASP.NET Core MVC. It combines server-side rendering with Razor views, Bootstrap styling, and an Entity Framework Core data access layer targeting a PostgreSQL database hosted through Supabase. The system is designed for student registration, event creation, competition booking, volunteer assignment, payment verification, notifications, and role-based access.

## Architecture Explanation

### Request lifecycle

1. Browser sends HTTP request.
2. ASP.NET Core routing maps it to a controller and action.
3. Controller uses `ApplicationDbContext` to query or update the database.
4. Controller prepares view model data and returns a Razor view.
5. Browser renders HTML and applies CSS/JavaScript.
6. Any dynamic updates are handled using JavaScript and Supabase realtime messages.

### Backend startup and configuration

- `Program.cs` loads configuration values from `appsettings.json` and environment variables.
- The database connection string is resolved from `ConnectionStrings:DefaultConnection`.
- `builder.Services.AddDbContext<ApplicationDbContext>(options => options.UseNpgsql(connectionString, npgsqlOptions => npgsqlOptions.CommandTimeout(120)))` configures PostgreSQL access.
- Authentication uses cookie-based login and logout paths.
- Authorization policies are defined for `OrganizerOnly`, `StudentOnly`, and `OrganizerOrAdmin`.
- Scoped services are registered for dashboard, ticket, notification, and Supabase realtime operations.

### Connection to Supabase/PostgreSQL

- The application connects to Supabase PostgreSQL through `Npgsql`.
- `appsettings.json` contains a Supabase pooler host, port `5432`, database `postgres`, username, and password.
- SSL mode is required and server certificate trust is enabled.
- The app also includes a Supabase realtime broadcast service configured by `Supabase:Url` and `Supabase:ServiceRoleKey`.

## Backend Implementation

### Controllers and endpoint mapping

- `AccountController` manages authentication, registration, login, and logout.
- `HomeController` serves the landing page.
- `EventController` handles event listing, details, creation, and deletion.
- `CompetitionController` handles competition listing, details, and creation.
- `RegistrationController` handles registration flows for individuals and teams.
- `PaymentController` handles payment proof uploads, payment listing, and verification.
- `NotificationController` handles listing notifications and marking them read.
- `RoleRequestController`, `RoleManagementController`, and staff assignment controllers are part of role workflows.

### How backend handles requests

- Each controller action receives request data from query strings, route values, or form POSTs.
- Validation is enforced through view model validation attributes, manual checks, and model state.
- Actions use Entity Framework Core methods like `FirstOrDefaultAsync`, `AnyAsync`, `Where`, `Include`, `Add`, `Remove`, `SaveChangesAsync`.
- Transactions are used in registration and deletion flows to maintain consistency.

### Connection string behavior

- The connection string is stored in `appsettings.json` and can be overridden by environment variables.
- The same connection string is used to create an `ApplicationDbContext` instance on each request.
- `Npgsql` options include a long command timeout of 120 seconds to account for Supabase pooler latency.
- The startup code executes a lightweight `SELECT 1` ping after migrations to warm up the database connection.

### Supabase/PostgreSQL integration

- The backend connects directly to PostgreSQL and does not call Supabase client libraries for query operations.
- Supabase is used as the hosting provider for PostgreSQL.
- Realtime broadcast uses the Supabase REST-like realtime API with the service role key.
- The `SupabaseRealtimeService` posts JSON payloads to `<SupabaseUrl>/realtime/v1/api/broadcast`.

### Primary keys and foreign keys

- `users.userid` is the primary key for `users`.

- `event.createdby` references `organizer.userid`.
- `competition.eventid` references `event.eventid`.
- `registration.competitionid` references `competition.competitionid`.
- `registration.userid` references `users.userid`.
- `registration.teamid` references `team.teamid`.
- `payment.registrationid` references `registration.registrationid`.
- `ticket.registrationid` references `registration.registrationid`.
- `team.leaderuserid` references `users.userid`.
- `team.competitionid` references `competition.competitionid`.
- `teammember.teamid` references `team.teamid`.
- `eventstaff.eventid` references `event.eventid` and `eventstaff.userid` references `volunteer.userid`.
- `competitionstaff.competitionid` references `competition.competitionid` and `competitionstaff.userid` references `volunteer.userid`.
- `notification.userid` references `users.userid`.
- `organizerrolerequest.studentid` references `student.userid` and `approvedby` references `admin.userid`.
- `volunteereventrequest.eventid` references `event.eventid`, `studentid` references `student.userid`, and reviewed-by columns reference `organizer.userid` / `admin.userid`.

### Junction tables

- `eventstaff` is a junction table representing the many-to-many relationship between `event` and volunteer users.
- `competitionstaff` is a junction table representing the many-to-many relationship between `competition` and volunteer users.
- `team` and `teammember` together model a team membership group, with `team` also related to `registration`.

### Unique keys and constraints

- `users.email` is unique via an index defined in `ApplicationDbContext` and enforced by the database.
- `student.rollnumber` is unique.
- `admin` has a unique singleton index to keep only one admin row.
- `volunteereventrequest(eventid, studentid)` has a unique partial index for pending requests.
- Several `CHECK` constraints exist in migrations: valid registration statuses and valid volunteer request statuses.

### ON DELETE CASCADE usage

- `admin`, `organizer`, `volunteer`, `student` use cascading delete from `users`.
- `eventstaff`, `competitionstaff`, `volunteereventrequest`, and `notification` cascade when parent rows are deleted.
- `volunteereventrequest` uses `ON DELETE SET NULL` for reviewed-by foreign keys.
- `Registration.TeamID` is configured with `OnDelete(DeleteBehavior.SetNull)` so deleting a team does not delete the registration.

## SQL Queries Explanation

### SELECT

Examples in code:

- `await _db.Users.AsNoTracking().FirstOrDefaultAsync(u => u.Email == model.Email)`
- `await _context.Events.AsNoTracking().Include(e => e.Creator).ThenInclude(c => c.User).FirstOrDefaultAsync(e => e.EventID == id)`
- `await _context.Registrations.AsNoTracking().Where(r => r.UserID == userId.Value).Include(r => r.Competition).ThenInclude(c => c.Event).ToListAsync()`
- `await _context.Notifications.AsNoTracking().Where(n => n.UserID == userId.Value).OrderByDescending(n => n.CreatedAt).ToListAsync()`

These queries produce SQL SELECT statements that retrieve entities and join related tables using EF Core `Include`.

### INSERT

Examples in code:

- ` _db.Users.Add(user); await _db.SaveChangesAsync();`
- `_db.Students.Add(new Student { ... }); await _db.SaveChangesAsync();`
- `_context.Events.Add(entity); await _context.SaveChangesAsync();`
- `_context.Registrations.Add(new Registration { ... }); await _context.SaveChangesAsync();`
- `_context.Payments.Add(payment); await _context.SaveChangesAsync();`
- `_context.Tickets.Add(new Ticket { ... }); await _context.SaveChangesAsync();`

EF Core translates these into PostgreSQL `INSERT` statements.

### UPDATE

Examples in code:

- `payment.Status = PaymentStatuses.Approved; payment.VerifiedBy = verifierId.Value; await _context.SaveChangesAsync();`
- `lockedCompetition.AvailableSeats = decision.UpdatedAvailableSeats; await _context.SaveChangesAsync();`
- `existingPayment.Status = PaymentStatuses.Pending; existingPayment.SubmittedAt = DateTime.UtcNow; await _context.SaveChangesAsync();`

These result in SQL `UPDATE` statements.

### DELETE

Examples in code:

- `_context.Registrations.RemoveRange(registrations);`
- `_context.TeamMembers.RemoveRange(members);`
- `_context.EventStaffAssignments.RemoveRange(eventStaff);`
- `_context.Events.Remove(eventEntity); await _context.SaveChangesAsync();`

These are translated into SQL `DELETE` statements or cascades where configured.

### Explicit raw SQL examples

- `await db.Database.ExecuteSqlRawAsync("SELECT 1");`
- `await db.Database.ExecuteSqlRawAsync("CREATE TABLE IF NOT EXISTS public.schema_migrations ...")`
- `await db.Database.ExecuteSqlRawAsync("INSERT INTO public.schema_migrations(filename, appliedat) VALUES ({0}, NOW())", filename)`
- `await _context.Database.ExecuteSqlInterpolatedAsync($"SELECT 1 FROM competition WHERE competitionid = {vm.CompetitionID} FOR UPDATE");`

The raw SQL is used for startup migration tracking and for acquiring a row lock before booking registration.

### Joins

The project uses EF Core `Include(...)` to produce SQL joins.

- `Include(e => e.Creator).ThenInclude(c => c.User)` joins `event` to `organizer` to `users`.
- `Include(r => r.Competition).ThenInclude(c => c.Event)` joins `registration`, `competition`, and `event`.
- `Include(p => p.Registration).ThenInclude(r => r.Competition)` joins `payment`, `registration`, and `competition`.

These are effectively inner joins in SQL.

### Subqueries

- `AnyAsync(...)` is implemented by SQL `EXISTS`.
- `Where(r => competitionIds.Contains(r.CompetitionID))` generates an `IN` clause.
- Migrations use `SELECT ... FROM pg_constraint ...` and `WHERE EXISTS (...)`.

### Join vs Subquery

- A JOIN combines rows from two tables into a single result set. Example: `Include(r => r.Competition).ThenInclude(c => c.Event)` joins registration, competition, and event.
- A SUBQUERY is a query inside another query. Example: `AnyAsync()` on `EventStaffAssignments` becomes an `EXISTS (...)` subquery when verifying volunteer permissions.

In this project, joins are used to load related entities, while subqueries are used to check existence, authorization, and constraints.

## Normalization & Anomalies

### 1NF

The tables are in first normal form because:

- Each table stores atomic values in each column.
- There are no repeating groups in any table.
- Examples: `users` stores one email per row, `competition` stores one location per row, `teammember` stores one email per member.

### 2NF

The tables are in second normal form because:

- Each non-key attribute is fully dependent on the primary key.
- Composite keys are present only in `eventstaff` and `competitionstaff`, and their attributes depend on the full composite key.
- Example: `eventstaff.Role` depends on both `eventid` and `userid`.

### 3NF

The tables are in third normal form because:

- No non-key column depends on another non-key column.
- Computed values are not stored; `Event.Status` is computed in code rather than persisted.
- The model avoids storing redundant user contact details in multiple tables.

### Why current schema is normalized

- `users` contains core identity and contact information.
- Role profiles (`student`, `organizer`, `volunteer`, `admin`) are separated into one-to-one tables.
- `registration`, `payment`, and `ticket` are separate entities linked by foreign keys.
- `team` and `teammember` separate groups from individual member details.

### Anomalies

- **Insertion anomaly:** A user can be created without a role-specific profile, but the registration process ensures a `student` profile is created immediately during signup. If a future role change is added without a profile, incomplete data could occur.
- **Update anomaly:** If user contact info were duplicated in a role-specific table, updating the email or phone in one place but not the other would cause inconsistency. The current design avoids this by keeping contact data in `users`.
- **Deletion anomaly:** Deleting an `event` cascades to competitions and related registrations, which is good. But if `registration` were deleted without cascading `payment` or `ticket`, orphan rows could appear. The code tries to remove dependent rows manually.

## Supabase & RLS

### How Supabase is used

- Supabase hosts the PostgreSQL database accessed by the backend.
- The connection string in `appsettings.json` points to a Supabase-hosted database.
- Supabase realtime is used for broadcasting seat updates and notifications.
- The backend sends JSON payloads to Supabase realtime channels using `SupabaseRealtimeService`.

### Data retrieval and storage

- Data retrieval is performed by Entity Framework Core against PostgreSQL.
- Data storage is performed by `SaveChangesAsync()` after creating, updating, or deleting entities.
- Supabase is not used as a direct JavaScript client in the current backend-driven application.

### API interaction with Supabase

- The broadcast API is invoked with bearer authentication using `ServiceRoleKey`.
- Messages are sent as JSON objects like `{ competitionId, availableSeats }` and `{ userId, message, createdAt }`.

### Row Level Security (RLS)

- There is no RLS policy code inside the ASP.NET application.
- `appsettings.json` includes blank Supabase keys for `AnonKey` and `ServiceRoleKey`, but these are not used for application authorization.
- The application enforces authorization through ASP.NET cookie auth and database role tables.

### Suggested RLS policies if implemented

- Allow `users` to update only their own profile rows.
- Allow `organizer` actions on events and competitions they created.
- Allow `student` access to their own registrations, payments, and notifications.
- Allow volunteers to view payments and tickets only for events/competitions they are assigned to.

### If Supabase goes down

- The backend will likely fail during database operations and return error pages or validation errors.
- The startup `SELECT 1` ping is ignored on failure, so the app can still start but will have runtime failures.
- The frontend will not be able to render updated pages correctly if PostgreSQL is unavailable.

### Improvements

- Add explicit retry and circuit-breaker logic for transient Supabase failures.
- Provide user-facing error pages when the database is unavailable.
- Add offline fallback or cached read state for non-critical pages.
- Implement proper RLS if migrating to direct Supabase client integration.

## API & Integration

### Frontend to backend integration

- The frontend uses Razor-generated HTML forms and standard HTTP POST requests.
- Form actions are directed to controller methods like `Account/Register`, `Event/Create`, `Registration/Create`, and `Payment/Create`.
- `ModelState.IsValid` is used to validate form data on the server.
- `TempData` and view models are used to pass success and error messages back to views.

### Backend processing of requests

- Controllers authenticate the user with `User.GetUserId()` and role claims.
- Data loading and updates are executed through the EF DbContext.
- Transactions are used during registration and deletion to avoid partial state.
- Business rules are enforced in controller logic, e.g. seat availability, role membership, date bounds.

### JSON communication

- The main frontend/backend communication is not JSON-based, but form-encoded POST data.
- Supabase realtime broadcast uses JSON payloads:
  - Seat update payload: `{ competitionId, availableSeats }`
  - Notification payload: `{ userId, message, createdAt }`
- The backend serializes these payloads with `JsonSerializer.Serialize`.

### Full integration step-by-step

1. User loads a page such as `Event/Index`.
2. Backend queries `event`, `competition`, and related user info.
3. The Razor view renders HTML and includes static CSS/JS.
4. The browser displays the page.
5. User submits a form to book a competition or upload proof.
6. The controller validates input and updates the database.
7. If relevant, the backend publishes a realtime broadcast to Supabase.
8. The browser receives the new page or redirect response.
9. If realtime is active, connected clients receive live updates and adjust DOM elements.

## Error Handling (Server Down Case)

### Current behavior

- Startup errors during migrations are logged but may not prevent the app from launching.
- `SELECT 1` database ping failures are ignored at startup.
- Controller errors are caught and translated into `TempData` error messages or standard HTTP responses.
- Realtime broadcast failures in `SupabaseRealtimeService` are swallowed to avoid breaking the main flow.

### How frontend behaves when backend/db is down

- The user may see an exception page or failed form submission.
- The application does not gracefully degrade to offline mode.
- Real-time notifications and seat updates will stop working if Supabase connectivity fails.

### Suggested improvements

- Show a dedicated error page when the database is unavailable.
- Use a circuit breaker around database operations.
- Display user-friendly messages such as `Service unavailable. Please try again later.`
- Add frontend retry logic for realtime subscriptions.

## Viva Questions & Answers

### What is a primary key vs a foreign key?

- A **primary key** uniquely identifies each row in a table. In this project, `users.userid` and `event.eventid` are examples.
- A **foreign key** is a column that references a primary key in another table. Example: `competition.eventid` references `event.eventid`.

### What is a unique key (non-primary)?

- A **unique key** enforces uniqueness without being the primary key. Example: `users.email` is indexed uniquely, meaning emails cannot repeat.

### What is a junction table?

- A junction table models a many-to-many relationship. Example: `eventstaff` links volunteers and events, while `competitionstaff` links volunteers and competitions.

### What are joins and their types?

- Joins combine rows from multiple tables. The main type used in this project is the inner join produced by EF Core `Include(...)`.
- Example: joining `registration` to `competition` and `event` to show a student’s registrations.

### What is the difference between JOIN and SUBQUERY?

- A **JOIN** combines rows from two tables in a single result set.
- A **SUBQUERY** runs a query inside another query, often used for existence checks.
- Example join: `Include(r => r.Competition).ThenInclude(c => c.Event)`.
- Example subquery: `AnyAsync()` becomes an `EXISTS` clause.

### What is normalization?

- Normalization is the process of organizing a database to reduce redundancy and dependency.
- 1NF requires atomic values.
- 2NF removes partial dependencies on composite keys.
- 3NF removes transitive dependencies.
- The project schema is close to 3NF because related data is separated into tables and foreign keys.

### What are anomalies?

- **Insertion anomaly:** inability to add valid data without other unrelated data.
- **Update anomaly:** redundant data requiring multiple updates.
- **Deletion anomaly:** deleting one row causes loss of other needed data.
- In this project, the schema avoids many anomalies through normalized design, but some risk remains if cross-table role data is altered incorrectly.

### SQL vs NoSQL

- SQL uses structured tables, rows, columns, relationships, and joins.
- NoSQL uses document, key-value, or graph models without fixed relational schema.
- This project is SQL-based and relies on relational constraints, foreign keys, and normalized tables.

### What is JSON format?

- JSON is an object-based data format with key/value pairs.
- In this project, JSON is used for realtime broadcast payloads to Supabase, not as the main form submission format.
- Example payload: `{ "competitionId": 5, "availableSeats": 12 }`.

### How does the API work?

- The project uses MVC controller actions as the API layer.
- Requests are routed to actions like `Registration/Create` and `Payment/Create`.
- Actions read request data, validate it, update the database, and return views or redirects.

### How is data retrieved from tables?

- `DbContext` queries return entity objects.
- Example: `await _context.Events.AsNoTracking().ToListAsync()` reads event rows.
- Related data is included with `Include(...)`.

### How does the frontend update dynamically?

- Pages update on server-side render after each request.
- Live updates occur using JavaScript that listens to Supabase broadcast channels.
- Example: seat counts on competition pages are updated without a full refresh when broadcast messages arrive.
