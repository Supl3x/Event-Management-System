1. Problem Statement
Current university campus systems suffer from fragmented, informal approaches to managing events. The following issues were identified through stakeholder analysis at NED University of Engineering & Technology:
No centralized database for event discovery, registration, or tracking.
Manual, error-prone payment verification via WhatsApp screenshots.
Inability to prevent duplicate registrations or enforce team size limits programmatically.
Loss of event data due to reliance on fragmented communication channels (social media, notice boards, chat groups).
No digital audit trail for tickets, payments, or organizer approvals.
Organizer and volunteer role management is entirely manual and informal.
These issues result in administrative overhead, data inconsistency, and a poor experience for students and faculty alike. A structured, normalized relational database system is required to replace these ad-hoc processes.
2. Project Description
2.1 Project Overview
The Smart Campus Event Ticketing & Real-Time Event Portal System is a centralized digital platform designed to manage all university events efficiently. It replaces the scattered approach of sharing event information over WhatsApp, social media, and physical notice boards with a single structured portal.
The system supports four distinct user roles: Students, Organizers, Volunteers, and Admins each with specific access rights and capabilities enforced at the database level through role-based constraints.
2.2 System Objectives
Provide a single, authoritative platform for all campus event information and registration.
Enforce business rules (team size limits, duplicate prevention, role validation) at the database layer.
Automate the ticket generation workflow: Registration → Payment Submission → Verification → QR Ticket.
Maintain a full digital audit trail of registrations, payments, verifications, and approvals.
Support role-based access with specialization (a student can also be an organizer).

3. Business Requirements
3.1 Functional Requirements
Users can register as Students; Admins can promote users to Organizer or Volunteer roles.
Organizers can create and manage Events and Competitions linked to those events.
Students can register for competitions (individually or as a team).
Each registration must be followed by a payment screenshot upload.
Organizers or Volunteers can approve or reject payment submissions.
Volunteers can be assigned to specific events or competitions with designated roles.
Admin can view all system data and manage role requests.
3.2 Non-Functional Requirements
Data Integrity: All constraints (UNIQUE, NOT NULL, CHECK, FK) enforced at the database level.
Scalability: Schema designed for horizontal scalability with PostgreSQL (Supabase).
Security: Role-level access control via Supabase Row Level Security (RLS) policies.
Availability: Cloud-hosted database ensures high uptime.
Performance: Indexes on all FK and frequently queried columns.

4. Current System Limitations
4.1 Overview of Existing System
Prior to this system, all event management at the university was handled informally. Organizers would post event details on WhatsApp groups, social media pages, and physical notice boards. Students registered by messaging an organizer directly or filling google/paper forms. Payments were made via bank transfer with screenshots shared on chat. Tickets, if issued at all, were verbal confirmations or PDF files sent manually.
4.2 Identified Limitations
Data Redundancy: Event and organizer information was repeated across multiple spreadsheets and messages with no single source of truth.
No Duplicate Prevention: A student could register for the same competition multiple times from different accounts.
Manual Payment Verification: Screenshots were matched against bank statements by hand, leading to errors and delays.
No Audit Trail: Deleted WhatsApp messages or lost files could permanently destroy registration history.
No Team Size Enforcement: Organizers had to manually count team members, leading to over-sized teams.
No Role Management: Any person could claim to be a volunteer or organizer with no system verification.
Fragmented Data: Event data lived across Google Forms, WhatsApp chats, and local Excel files — none integrated.
4.3 Impact of Limitations
These limitations caused significant issues: students were unable to verify their registration status, payments were occasionally lost or duplicated, and event organizers had no reliable way to generate attendance lists or enforce competition rules. The absence of a centralized system also made it impossible to produce analytical reports on participation trends or volunteer deployment.

9. Normalization Process
Normalization is the process of organizing a database to reduce redundancy and improve data integrity. It is applied in progressive steps: 1NF removes repeating groups; 2NF eliminates partial dependencies; 3NF removes transitive dependencies; BCNF ensures every determinant is a candidate key.
9.1 Un-Normalized Form (UNF)
The single flat table stores all system data together. It violates 1NF because the following columns contain multiple values in a single cell:
Organizer_Names, Organizer_Emails
Competition_Names, Competition_Descriptions, Competition_Locations, Competition_StartDates, Competition_EndDates, Competition_MaxTeamSizes, Competition_EntryFees
Team_Names, TeamMember_Names, TeamMember_RollNumbers, TeamMember_Departments, TeamMember_Emails
Registration_Types, Registration_Statuses, Payment_Screenshots, Pay_Statuses, Pay_VerifiedBy, Ticket_QRCodes, Ticket_UniqueCodes
Group A — Event & Organizer (multi-valued Organizer columns)
Group B — Competition (multi-valued columns per Event row)
Group C — Team & TeamMember (multi-valued per Competition row)
Group D — Registration, Payment & Ticket (multi-valued per Team row)
9.2 Anomalies in UNF
Insertion Anomaly
An insertion anomaly occurs when a valid piece of information cannot be stored without adding unrelated data. In the flat table, a new organizer or volunteer cannot be recorded unless they are already linked to an event, competition, team, payment, and ticket.
Example: Organizer "Dr. Nadia" (nadia@ned.edu) has just been granted organizer access but has not yet created any event. Row 3 below cannot be inserted because all non-organizer columns are empty — this is the Insertion Anomaly.
Deletion Anomaly
A deletion anomaly occurs when deleting a record unintentionally destroys other valuable data.
Example: If Team "Main Akela he kaafi hun" (the only team in Registration ID 3) is cancelled and its row is deleted, the associated Payment screenshot and QR-coded Ticket (UniqueCode: 198935D4) are permanently lost, even though the competition itself still exists.
Updation Anomaly
An update anomaly occurs when redundant data requires the same change to be made in multiple rows. Missing even one occurrence leaves the database inconsistent.
Example: If the venue of TechFest 2026 changes from "Department" to "Main Auditorium", every row in the unnormalized table that references TechFest 2026 must be updated. Since team member data causes the event to repeat for each member, missing even one row creates conflicting location values.
9.3 First Normal Form (1NF)
First Normal Form requires each column to hold exactly one atomic value. All multi-valued (semicolon-separated) cells are split into individual rows.
1NF Table — Representative Rows (each row now holds one atomic value per cell)
Observation: The 1NF table achieves atomic values per cell. However, massive redundancy remains: Event_Name, Competition_Name, Pay_Status repeat identically across rows. Many non-key attributes depend only on part of the composite candidate key, a partial dependency addressed in 2NF.
9.4 Second Normal Form (2NF)
Second Normal Form requires that every non-key attribute depends on the entire candidate key, not just part of it. Partial dependencies are identified and resolved by decomposition.
Event_Location, Event_StartDate, Event_EndDate → depend only on Event_Name
Competition attributes → depend only on Competition_Name
TeamMember details (Name, Department, Email) → depend only on Member_RollNo
Team attributes → depend only on Team_Name
Payment & Ticket data → depend only on the Registration, not the full composite key

9.5 Third Normal Form (3NF)
Third Normal Form requires that no non-key attribute is transitively dependent on the primary key. The following transitive dependencies from 2NF are resolved:
Users: UserID → Role → (RollNumber, Department) — resolved by splitting into sub-tables: Student, Organizer, Admin, Volunteer.
Event: EventID → CreatedBy_Name → CreatedBy_Email — resolved by replacing embedded name/email with FK to Organizer.
Team: TeamID → LeaderName → LeaderEmail — resolved by replacing with LeaderUserID (FK → Users).
Registration: RegistrationID → UserName → (Email, Phone) — resolved by using UserID FK.
Payment: PaymentID → VerifierName → VerifierEmail — resolved by using VerifiedBy UserID FK.

9.6 Boyce-Codd Normal Form (BCNF)
A table is in BCNF if for every non-trivial functional dependency X → Y, X must be a superkey. BCNF is stricter than 3NF, it addresses anomalies from non-trivial FDs where the determinant is not a candidate key.

13. Limitations
13.1 Technical Limitations
The system currently supports a single university domain. Multi-campus support would require additional tenant-based schema changes.
Supabase's free tier has storage and row limits that may be reached during large-scale university-wide deployment.
QR code generation uses UUID-based values; a dedicated QR library integration is not yet implemented in the database layer.
The schema does not yet include full-text search indexes for event/competition discovery by keyword.
13.2 Design Limitations
The current schema includes a WAITLIST_ENTRY table with priority-number ordering, but automatic promotion logic (moving a waitlisted registration to confirmed when a seat opens) is not yet implemented as a database trigger.
Payment supports only screenshot-based verification. Integration with a payment gateway (JazzCash, EasyPaisa) is outside the current scope.
The Ticket entity generates a static QR code at approval time. Dynamic QR scanning for attendance tracking is not implemented.
13.3 Time & Resource Constraints
The project was completed within a single semester with a team of four students. This limited the scope to core normalization, schema design, and sample queries. A full front-end integration with ASP.NET Core and live Supabase deployment was designed but not fully implemented within the project timeline.
14. Future Enhancements
14.1 Short-Term Enhancements
Implement an automatic waitlist-promotion trigger so that when a confirmed registration is cancelled, the highest-priority waitlisted registration is automatically upgraded and the student is notified.
Add email/SMS notification triggers to inform students of payment approval and ticket generation.
Build a reporting module with pre-defined analytical queries accessible to admins.
Add full-text search on Event and Competition names using PostgreSQL tsvector indexes.
14.2 Long-Term Enhancements
Integrate a payment gateway (JazzCash / EasyPaisa) to replace manual screenshot verification.
Develop a dynamic QR scanning mobile app for real-time attendance tracking at events.
Extend the schema to support multi-campus deployments with tenant-based row-level security.
Build a public event portal front-end using ASP.NET Core / Razor Pages connected to Supabase.
15. Conclusion
The Smart Campus Event Ticketing & Real-Time Event Portal System database has been successfully designed and normalized to Boyce-Codd Normal Form (BCNF). The schema consists of 18 tables covering all required entities, relationships, and business rules specified in the project overview, drawn from a thorough analysis of the real-world campus event management problem at NED University.
The normalization process began with a single large unnormalized (UNF) relation and proceeded step-by-step through 1NF (elimination of repeating groups), 2NF (removal of partial dependencies), 3NF (elimination of transitive dependencies by separating the User super-entity into sub-tables), and finally we tried BCNF (resolving the RollNumber → Department FD in TeamMember). Each step was demonstrated with concrete sample data drawn from the actual system.
Key achievements of this design include: full normalization to BCNF eliminating all insertion, deletion, and update anomalies; correct EER specialization with overlapping and partial participation; three many-to-many relationships modeled via junction tables; ten or more business rules enforced at the database layer through triggers and CHECK constraints; a complete DDL and DML SQL script ready for one-click deployment on Supabase; and comprehensive sample analytical queries covering joins, aggregations, subqueries, and NOT EXISTS patterns.
The complete source code and SQL scripts are available at: github.com/EventHub-NED

EER Specialization Summary

USER is the super-entity (base class).
Sub-entities: STUDENT, ORGANIZER, VOLUNTEER, ADMIN.
Type: OVERLAPPING — a user can simultaneously hold multiple roles (e.g., a Student can also become an Organizer). Each active role has a row in its respective sub-table sharing the same UserID.
Participation: PARTIAL — not every user must belong to a subtype.
Implementation: Each sub-entity shares the UserID primary key with the parent (table-per-hierarchy pattern).

BCNF Result: The final schema is in BCNF. Every functional dependency X → Y has X as a superkey (candidate key). All determinants are candidate keys. No update, insertion, or deletion anomalies exist.
-- 1. BASE USER TABLE (Super-entity)
CREATE TABLE Users (
  UserID  SERIAL PRIMARY KEY,
  Name    VARCHAR(150) NOT NULL,
  Email   VARCHAR(255) NOT NULL UNIQUE,
  Phone   VARCHAR(20)  NOT NULL
);

-- 2. STUDENT (Specialization of User)
CREATE TABLE Student (
  UserID      INT PRIMARY KEY REFERENCES Users(UserID) ON DELETE CASCADE,
  RollNumber  VARCHAR(50)  NOT NULL UNIQUE,
  Department  VARCHAR(100) NOT NULL
);

-- 3. ADMIN, ORGANIZER, VOLUNTEER
CREATE TABLE Admin     ( UserID INT PRIMARY KEY REFERENCES Users(UserID) ON DELETE CASCADE );
CREATE TABLE Organizer ( UserID INT PRIMARY KEY REFERENCES Users(UserID) ON DELETE CASCADE );
CREATE TABLE Volunteer ( UserID INT PRIMARY KEY REFERENCES Users(UserID) ON DELETE CASCADE );

-- 4. ORGANIZER ROLE REQUEST
CREATE TABLE OrganizerRoleRequest (
  RequestID   SERIAL PRIMARY KEY,
  StudentID   INT NOT NULL REFERENCES Student(UserID) ON DELETE CASCADE,
  ApprovedBy  INT REFERENCES Admin(UserID) ON DELETE SET NULL,
  Status      VARCHAR(20) NOT NULL DEFAULT 'Pending'
              CHECK (Status IN ('Pending', 'Approved', 'Rejected')),
  RequestedAt TIMESTAMP NOT NULL DEFAULT NOW(),
  ReviewedAt  TIMESTAMP
);

-- 5. EVENT
CREATE TABLE Event (
  EventID    SERIAL PRIMARY KEY,
  Name       VARCHAR(200) NOT NULL,
  Department VARCHAR(100) NOT NULL,
  Location   VARCHAR(200) NOT NULL,
  StartDate  DATE NOT NULL,
  EndDate    DATE NOT NULL,
  CreatedBy  INT  NOT NULL REFERENCES Organizer(UserID) ON DELETE RESTRICT,
  CONSTRAINT chk_event_dates CHECK (EndDate >= StartDate)
);

-- 6. COMPETITION
CREATE TABLE Competition (
  CompetitionID SERIAL PRIMARY KEY,
  EventID       INT  NOT NULL REFERENCES Event(EventID) ON DELETE CASCADE,
  Name          VARCHAR(200) NOT NULL,
  Description   TEXT,
  Location      VARCHAR(200) NOT NULL,
  StartDate     DATE NOT NULL,
  EndDate       DATE NOT NULL,
  MaxTeamSize   INT  NOT NULL DEFAULT 1 CHECK (MaxTeamSize BETWEEN 1 AND 5),
  EntryFee      NUMERIC(10,2) NOT NULL DEFAULT 0.00,
  CONSTRAINT chk_comp_dates CHECK (EndDate >= StartDate)
);
-- 7. TEAM
CREATE TABLE Team (
  TeamID        SERIAL PRIMARY KEY,
  TeamName      VARCHAR(200) NOT NULL,
  LeaderUserID  INT NOT NULL REFERENCES Users(UserID) ON DELETE RESTRICT,
  CompetitionID INT NOT NULL REFERENCES Competition(CompetitionID) ON DELETE CASCADE,
  CONSTRAINT uq_team_competition UNIQUE (TeamName, CompetitionID)
);

-- 8. TEAM MEMBER (BCNF fix: MemberID sole PK)
CREATE TABLE TeamMember (
  MemberID   SERIAL PRIMARY KEY,
  TeamID     INT NOT NULL REFERENCES Team(TeamID) ON DELETE CASCADE,
  Name       VARCHAR(150) NOT NULL,
  RollNumber VARCHAR(50),
  Department VARCHAR(100),
  Email      VARCHAR(255) NOT NULL,
  CONSTRAINT uq_team_member_email UNIQUE (TeamID, Email)
);

-- 9. REGISTRATION
CREATE TABLE Registration (
  RegistrationID SERIAL PRIMARY KEY,
  CompetitionID  INT NOT NULL REFERENCES Competition(CompetitionID) ON DELETE RESTRICT,
  UserID         INT NOT NULL REFERENCES Users(UserID) ON DELETE RESTRICT,
  TeamID         INT REFERENCES Team(TeamID) ON DELETE SET NULL,
  Type           VARCHAR(20) NOT NULL CHECK (Type IN ('Individual','Team')),
  Status         VARCHAR(20) NOT NULL DEFAULT 'Pending'
                 CHECK (Status IN ('Pending','Approved','Rejected')),
  RegisteredAt   TIMESTAMP NOT NULL DEFAULT NOW(),
  CONSTRAINT uq_user_competition UNIQUE (UserID, CompetitionID)
);

-- 10. PAYMENT (1:1 with Registration)
CREATE TABLE Payment (
  PaymentID      SERIAL PRIMARY KEY,
  RegistrationID INT NOT NULL UNIQUE REFERENCES Registration(RegistrationID) ON DELETE CASCADE,
  Screenshot     VARCHAR(500) NOT NULL,
  Status         VARCHAR(20) NOT NULL DEFAULT 'Pending'
                 CHECK (Status IN ('Pending','Approved','Rejected')),
  VerifiedBy     INT REFERENCES Users(UserID) ON DELETE SET NULL,
  SubmittedAt    TIMESTAMP NOT NULL DEFAULT NOW(),
  VerifiedAt     TIMESTAMP
);

-- 11. TICKET (Auto-generated on payment approval)
CREATE TABLE Ticket (
  TicketID       SERIAL PRIMARY KEY,
  RegistrationID INT  NOT NULL UNIQUE REFERENCES Registration(RegistrationID) ON DELETE CASCADE,
  QRCode         TEXT NOT NULL UNIQUE DEFAULT gen_random_uuid()::TEXT,
  UniqueCode     VARCHAR(50) NOT NULL UNIQUE
                 DEFAULT UPPER(SUBSTRING(gen_random_uuid()::TEXT, 1, 8)),
  GeneratedAt    TIMESTAMP NOT NULL DEFAULT NOW()
);

CREATE OR REPLACE FUNCTION auto_generate_ticket()
RETURNS TRIGGER AS $$
BEGIN
  IF NEW.Status = 'Approved' AND OLD.Status <> 'Approved' THEN
    INSERT INTO Ticket (RegistrationID)
    VALUES (NEW.RegistrationID)
    ON CONFLICT (RegistrationID) DO NOTHING;
  END IF;
  RETURN NEW;
END; $$ LANGUAGE plpgsql;

CREATE TRIGGER trg_auto_ticket
AFTER UPDATE OF Status ON Payment
FOR EACH ROW EXECUTE FUNCTION auto_generate_ticket();
CREATE OR REPLACE FUNCTION check_team_size()
RETURNS TRIGGER AS $$
DECLARE current_size INT; max_size INT;
BEGIN
  SELECT COUNT(*) INTO current_size FROM TeamMember WHERE TeamID = NEW.TeamID;
  SELECT c.MaxTeamSize INTO max_size
  FROM Team t JOIN Competition c ON t.CompetitionID = c.CompetitionID
  WHERE t.TeamID = NEW.TeamID;
  IF current_size >= max_size THEN
    RAISE EXCEPTION 'Team size limit exceeded. Max allowed: %', max_size;
  END IF;
  RETURN NEW;
END; $$ LANGUAGE plpgsql;

CREATE TRIGGER trg_team_size_limit
BEFORE INSERT ON TeamMember
FOR EACH ROW EXECUTE FUNCTION check_team_size();
CREATE OR REPLACE FUNCTION check_payment_verifier()
RETURNS TRIGGER AS $$
BEGIN
  IF NEW.VerifiedBy IS NOT NULL THEN
    IF NOT EXISTS (
      SELECT 1 FROM Organizer WHERE UserID = NEW.VerifiedBy
      UNION
      SELECT 1 FROM Volunteer WHERE UserID = NEW.VerifiedBy
    ) THEN
      RAISE EXCEPTION 'Payment can only be verified by an Organizer or Volunteer';
    END IF;
  END IF;
  RETURN NEW;
END; $$ LANGUAGE plpgsql;

CREATE TRIGGER trg_payment_verifier
BEFORE INSERT OR UPDATE ON Payment
FOR EACH ROW EXECUTE FUNCTION check_payment_verifier();
